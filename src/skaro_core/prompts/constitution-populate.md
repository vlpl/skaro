# Заполнение Конституции из принятых требований

## Задача
Доработать Конституцию проекта в соответствии с принятыми требованиями.

## Входные данные
1. **Принятые требования** — список требований со статусом `accepted`
2. **Текущая Конституция** — существующий текст Конституции (может быть пустым)
3. **Контекст проекта** — название и описание проекта

## Правила обновления Конституции

### Что включать в Конституцию:
- ✅ **Платформы** — языки программирования, фреймворки, рантаймы
- ✅ **Программные продукты** — базы данных, очереди сообщений, кэши, внешние сервисы
- ✅ **Инструменты разработки** — системы контроля версий, CI/CD, линтеры, форматтеры
- ✅ **Стандарты** — соглашения о коде, тестировании, документации
- ✅ **Архитектурные принципы** — паттерны, ограничения, правила
- ✅ **Неизменные правила** — то, что нельзя менять без веских причин

### Что НЕ включать в Конституцию:
- ❌ Функциональные требования (это не архитектурные решения)
- ❌ Временные решения (то, что может измениться)
- ❌ Детали реализации конкретных фич
- ❌ Бизнес-логика предметной области

## Инструкция для LLM

Проанализируй принятые требования и определи:

1. **Технологический стек** — какие платформы, языки, фреймворки упоминаются как обязательные
2. **Инфраструктурные компоненты** — базы данных, кэши, очереди, которые требуются
3. **Архитектурные ограничения** — паттерны, принципы, которые диктуются требованиями
4. **Стандарты качества** — требования к тестированию, безопасности, производительности

Обнови Конституцию следующим образом:

1. **Сохрани существующие разделы**, которые актуальны
2. **Добавь новые разделы** на основе выявленных технологических требований
3. **Уточни формулировки** чтобы они соответствовали принятым требованиям
4. **Убери противоречия** между требованиями и текущей Конституцией
5. **Сфокусируйся на неизменном** — только то, что будет действовать на протяжении всего проекта

## Формат ответа

Верни **полный текст обновлённой Конституции** в формате Markdown, готовый к сохранению в файл `CONSTITUTION.md`.

## ⚠️ КРИТИЧЕСКИ ВАЖНО: Формат заголовков разделов

**Все заголовки разделов должны быть двуязычными:**
- **Формат:** `## {English Name} ({Русское название})`
- **Примеры:**
  - `## Technology Stack (Технологический стек)`
  - `## Coding Standards (Стандарты разработки)`
  - `## Testing (Тестирование)`
  - `## Constraints and Infrastructure (Ограничения и инфраструктура)`
  - `## Security (Безопасность)`
  - `## LLM Rules (Правила работы с LLM)`

**Подразделы также двуязычные:**
- `### {English Subsection} ({Русское название})`
- Пример: `### Programming Languages (Языки программирования)`

Это требование обязательно для корректной работы валидации Конституции!

## Структура Конституции:

```markdown
# Constitution of {Project Name} (Конституция проекта {Название})

## 1. Technology Stack (Технологический стек)
### 1.1 Programming Languages (Языки программирования)
### 1.2 Frameworks and Platforms (Фреймворки и платформы)
### 1.3 Databases and Storage (Базы данных и хранилища)
### 1.4 Infrastructure Components (Инфраструктурные компоненты)

## 2. Coding Standards (Стандарты разработки)
### 2.1 Code Style (Стиль кода)
### 2.2 Testing Requirements (Требования к тестированию)
### 2.3 Documentation (Документирование)

## 3. Architectural Principles (Архитектурные принципы)
### 3.1 Design Patterns (Паттерны проектирования)
### 3.2 Constraints (Ограничения)
### 3.3 Invariants (Инварианты)

## 4. Security (Безопасность)
### 4.1 Security Requirements (Требования к безопасности)
### 4.2 Authorization and Authentication (Авторизация и аутентификация)

## 5. LLM Rules (Правила работы с LLM)
### 5.1 LLM Usage Guidelines (Руководство по использованию LLM)
### 5.2 Automation (Автоматизация)

## 6. Inviolable Rules (Неприкосновенные правила)
Перечень правил, которые нельзя нарушать без пересмотра Конституции.
```

## Критерии качества

- ✅ Все технологии, упомянутые в принятых требованиях, отражены в Конституции
- ✅ Нет противоречий между требованиями и Конституцией
- ✅ Стилистика единая, текст читаемый
- ✅ Разделы структурированы по аналогии с другими фазами Skaro
- ✅ Акцент на неизменных решениях, а не временных
